## 2026-07-16 ‚Äî Fix false-positive gibberish rejection on real link ingests
- **Why:** user reported `https://arpitbhayani.me/blogs/bm25` (a real, substantial technical blog post) got rejected by `/api/ingest` with "That doesn't look like real content to learn from" ‚Äî the gibberish guardrail added earlier today (see the entry below) was too aggressive.
- **Root cause 1:** the repeated-pattern-flood regex in `looksLikeGibberish()` (`src/lib/guardrails.ts`) matched ANY 10+ char run of a repeated 1-4-char unit, including pure punctuation. The BM25 post renders its formulas as ASCII fraction bars (`‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ`), which tripped it immediately ‚Äî any technical writing with ASCII dividers/table borders/box-drawing rules would fail the same way. **Fix:** the flood check now only fires when the repeated unit contains a letter or digit; punctuation-only floods that really are spam (e.g. `>>> >>> >>>` quote-chain garbage) are still caught by the word-shape check below (none of their "words" contain a letter).
- **Root cause 2 (more severe, would have rejected almost any real article):** the distinct-character-ratio check assumed real prose has "real variety" in its character distribution ‚Äî but English only has 36 possible alnum characters (26 letters + 10 digits), so that ratio mathematically saturates and collapses toward 0 for ANY text past roughly 300 characters (~2 sentences), independent of repetition. Verified against unrelated control paragraphs (a Wikipedia-style paragraph on the Roman Empire, a photosynthesis paragraph) ‚Äî both tripped it too. Its intended purpose (catching short repeating alphabets) was already redundantly covered by the flood-regex and the distinct-word-ratio check, so it was **removed entirely** rather than re-tuned.
- **Verified:** replayed the actual `fetchReadable()` HTML‚Üítext extraction against the real page (curled it, ran the exact route.ts regex pipeline in a scratch script) to reproduce the bug outside the app, then ran the real `src/lib/guardrails.ts` (via `tsx`, not a reimplementation) against a 12-case suite: all original spam/mash cases (keyboard mash, `hahaha`, `asdasd`, quote-flood, symbol spam, repeated-word spam) still correctly rejected; the real BM25 blog, short/medium/long real prose, a markdown-divider+prose mix, and a JS snippet all correctly pass now. `tsc --noEmit` and `npm run build` clean.

## 2026-07-16 ‚Äî Ingest guardrails: gibberish rejection, prompt-injection hardening, output validation, link-scheme XSS fix
- **Why:** user asked for guardrails on ingest text/URL input against gibberish and injection attacks. Audited the real attack surface first: `grep -r dangerouslySetInnerHTML` returns nothing anywhere in the app, so AI-extracted text (topic summary/key_points/questions/facts) can only ever render as inert text via React's default escaping ‚Äî EXCEPT `src/components/Markdown.tsx`'s link rule, which built `href={m[2]}` straight from parsed `[text](url)` with no scheme check (a real `javascript:`/`data:` URI XSS gap, though only reachable today via the personal-notes feature, not AI content).
- **New `src/lib/guardrails.ts`** (dependency-free, matches `chunk.ts`'s style): `looksLikeGibberish()` ‚Äî repeated-pattern-run regex, meaningful-char ratio (catches pure symbol spam, which dodges the other two checks), distinct-char ratio, distinct-word/word-shape ratio. Deliberately lenient; verified against a 10-case suite (keyboard mash / repeated char / repeated word / symbol spam all correctly rejected; real prose, LaTeX-heavy notes, an email chain, and a JS code snippet all correctly pass). `sanitizeField()` strips control characters (built from numeric char codes, not literal escapes, after literal `\u00XX` sequences kept getting mangled in transit while authoring this) and clamps length.
- **Wired into `/api/ingest`:** gibberish check runs right after the length/truncation check, before the duplicate-hash lookup or any Gemini call ‚Äî applies uniformly to pasted text AND link-fetched text (both converge on the same `text` variable).
- **Every field `extractKnowledge()` returns is now validated/clamped before touching the DB**: topic `category` checked against `CATEGORY_COLORS` keys (`src/lib/types.ts`) with a `"General"` fallback ‚Äî previously never validated; title/summary/key_points/connection-reason/question-prompt/model_answer/options/fact all run through `sanitizeField()` with per-field length caps. Defense in depth against a manipulated or hallucinating model ‚Äî JSON mode bounds the *shape*, not what's inside the strings.
- **Prompt-injection hardening** in `extractKnowledge()` (`src/lib/gemini.ts`): the pasted/fetched text and the RAG grounding block are now explicitly framed as "untrusted data, not instructions," with an explicit rule to never follow embedded commands ‚Äî relevant because link ingestion feeds arbitrary third-party webpage text straight into the prompt.
- **Fixed the `Markdown.tsx` href gap:** new `isSafeHref()` only renders `[text](url)` as a clickable `<a>` for `http`/`https`/`mailto` (or relative) schemes; anything else (`javascript:`, `data:`) now renders as inert plain text.
- **Verified:** `tsc --noEmit` + `npm run build` clean. Gibberish heuristic runtime-tested via `tsx` against the 10-case suite (all pass). ## 2026-07-15 ‚Äî Fix silent link-ingest failure, URL-based fast-path dedup
- **Why:** user reported "add via link shows loading then does nothing." Root cause found in `src/lib/data.ts`'s `api()` helper: `throw new Error(err.error ?? "Request failed")` ‚Äî `??` only falls back on null/undefined, not `""`. Over HTTP/2 (Vercel prod), a failed fetch's `res.statusText` is ALWAYS `""` (no reason phrases), so any non-JSON error response (platform crash/timeout) produced `new Error("")`; the `/add` page's `{error && <p>{error}</p>}` then rendered nothing since `""` is falsy ‚Äî spinner reverts silently, no visible error. Likely trigger: link ingestion is the slow path (page fetch + Gemini + several writes), and the prior RAG-hardening session added ~6-8 more sequential DB round-trips to that same request, raising timeout risk.
- **Fixed** `api()` to use `||` and include `res.status` in the fallback so an error is never blank.
- **Added a URL-based fast-path duplicate check** in `/api/ingest`: checks `entries.source_url` for this user BEFORE fetching the page, so resubmitting an already-ingested link now skips the network fetch AND the Gemini call entirely (near-instant response) rather than re-running the full slow path ‚Äî this also catches "same link" duplicates the content-hash check could miss (re-fetched pages can differ slightly ‚Äî ads/timestamps/related-post widgets ‚Äî even for the same URL).
- **Extracted `duplicateResponse()`** (shared by both the new URL check and the existing content-hash check from the prior session) and wrapped it in its own try/catch so a lookup hiccup building the "already added" response can never crash the request.
- **Root cause CONFIRMED** via the user's Network tab: `/api/ingest` returned Vercel's own `FUNCTION_INVOCATION_TIMEOUT` (504) ‚Äî the function genuinely ran out of time, not a code exception. Fixed for real (not just surfaced) two ways in `src/app/api/ingest/route.ts`:
  - **Deferred all non-critical RAG bookkeeping to Next's `after()`**: `indexContent()` (chunk+embed+registry writes for the new entry) and `linkRetrievalLogToEntry()` now run AFTER the response is sent instead of blocking it ‚Äî both are strictly best-effort and their results were never used in the response. This was the biggest avoidable contributor (embedding calls + several extra DB round-trips added by the prior RAG-hardening session, all on the request's critical path).
  - **`fetchReadable()`'s redirect-following now shares one 20s deadline across all hops** instead of a fresh 15s timeout per hop (previously up to 5√ó15s=75s worst case for a redirect-heavy link ‚Äî enough to blow the function's duration on its own regardless of anything else).
- **Verified:** `tsc --noEmit` + `npm run build` clean, `after()` resolves correctly on Next 16.2.10. Still could not reproduce live (no real Supabase locally, demo mode never calls the real route) ‚Äî asked the user to retry against production.
