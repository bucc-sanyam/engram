## 2026-07-15 — Attempted-tracking bug fix, security review, Gemini fallback, RAG pipeline, DB decision
- **"Answered questions show as not attempted" + reroute bug FIXED.** Two root causes: (1) UTC-vs-local day mismatch — `/api/plan` done-tracking, the calendar and the heatmap all bucketed by `created_at.slice(0,10)` (UTC), so an evening-IST review landed on the next UTC day and never marked its plan item done. New `src/lib/dates.ts` centralises day math around the USER'S LOCAL day; client passes tz offset to `/api/plan?tz=`, quiz `finish`/plan `complete`; all client bucketing uses `localDayKey`. (2) The review page read `?topic=` once via `window.location.search` on mount, so clicking a different plan row (same `/review` pathname) never re-ran init → stale session reopened the old question. Now uses `useSearchParams` (Suspense-wrapped) with a full state reset per `topic` change. Also: `doneTopicsToday()` now unions `reviews` rows AND `topics.last_reviewed_at` (safety net if a reviews insert fails); full-plan runs sort pending-first (never open on an "already reviewed" card); a last-pending single-task deep link now closes the day itself; streak advances on every attempted session (was gated behind XP, which is retired → 0).
- **Demo progress persists** to `localStorage` (`engramia.demo.v1`) — guest reviews/reports/streak survive reload/hard-nav; `done`/`completed` reset on a new local day like the real backend.
- **Security review (whole codebase).** Added an **SSRF guard** to link ingest (`assertPublicHttpUrl`: http(s)+standard ports only; blocks loopback/private/link-local/CGNAT IPv4, IPv6 ULA/link-local, `localhost`/`.local`/`.internal`, creds-in-URL; redirects followed MANUALLY 5-hop so each hop is re-checked). **Per-user daily ingest cap** (`INGEST_DAILY_LIMIT`, default 20) + 120k-char store cap. All API routes now guard `request.json()` (malformed→400); `/auth/callback` redirects to `/login?error=auth` on session-exchange failure. Confirmed RLS covers every table incl. new `knowledge_chunks`. No XSS sinks, no committed secrets (`.env.local` gitignored).
- **Gemini model fallback** (`gemini.ts`): 429/quota or 503/overload retries down a chain (`GEMINI_MODEL` → `GEMINI_MODEL_FALLBACKS`, default flash → flash-lite → 2.0-flash → 2.0-flash-lite); a failed model cools down 5 min per warm instance.
- **RAG pipeline** (source-of-truth grounding). New `supabase/schema-rag.sql` (pgvector, `knowledge_chunks`, HNSW cosine index, `match_knowledge_chunks` RPC, RLS). `chunk.ts` (overlapping ~350-token chunker), `embedTexts()` in gemini.ts (`gemini-embedding-001`, 768-dim, L2-normalised, doc/query task types), `rag.ts` (`indexContent`/`indexAnswers`/`retrieveContext`/`formatContext`). Ingest retrieves the user's prior related chunks → grounds `extractKnowledge`, then indexes the new entry; quiz finish indexes typed answers. Everything best-effort: dormant + no errors until the SQL is run. Blog long-form-from-RAG deliberately deferred (would add an AI call per page view).
- **DB decision: STAY on Supabase, don't migrate to TiDB.** TiDB Cloud gives 5GB (vs 500MB) but has NO auth and no Postgres RLS — both load-bearing here (browser-direct queries rely on RLS; whole app is `@supabase/ssr`+auth). Migration = rebuild auth + hand-roll per-user isolation + full API-route rewrite + pgvector→TiDB-vector port. 500MB is workable for ~50 users at 768-dim; the real free-tier gotcha is the 7-day inactivity pause. Path: upgrade to Supabase Pro ($25/mo → 8GB, no pause, backups) when needed — far cheaper than migrating.
- **Verified:** tsc + prod build clean (16 routes; `/review` still prerenders static behind Suspense). In browser (demo): single-task deep link → answer → 80% report card; nav Review → full 10-item plan pending-first; answer d1 → hard reload → "9 remaining" + streak persisted + d1 done; click d3 → d3's question (no reroute to d1); no console errors. RAG untestable in demo (server/real-mode only) but build+typecheck green and graceful-degradation verified by design. graphify updated (461 nodes).

## 2026-07-13 — Fix /profile infinite loading (missing profiles row)
- **Symptom:** /profile stuck on "Loading…" on the real-Supabase build. Schema *file* was fine — `supabase/schema.sql` already had all profile columns + `handle_new_user` trigger (committed together in `2b3ec32`). Root cause: `getProfile()` did `.single()` and returned `data as Profile` ignoring the error, so a *missing profiles row* (account predates the trigger, or trigger never installed in that project) returned `null` → profile stayed null → hung. Page's `.catch(()=>{})` hid the error.
- **Fix:** `getProfile()` (`src/lib/data.ts`) now uses `.maybeSingle()`, throws on real errors, and **self-heals**: if no row, upserts one (`user_metadata.name` → email prefix). `/profile` page now shows an error card (with "run schema.sql" hint) instead of infinite Loading.
- **Backfill for existing users:** the trigger only fires on new signups, so pre-existing accounts still need a row (self-heal handles it on next load, or run the SQL insert-from-auth.users backfill).
- **Verified:** tsc clean; /profile still renders in demo mode (Demo Learner / LV 5, no regression). Real-Supabase self-heal not exercisable locally (env is demo-only).
