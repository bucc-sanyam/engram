## 2026-07-18 — Server-side security audit + fixes: SSRF DNS-rebinding, quiz Gemini-quota abuse, curl-hardening
- **Why:** user asked for a server-side security review, then asked to fix the two most serious findings, plus explicitly wanted the API hardened against a "curl in a terminal" style direct-access attack (no browser, no session).
- **Audit covered:** all 4 API routes' auth checks, RLS on every table across all 4 schema files (all correct, all present, including proper join-based checks for junction tables), secrets handling (no service-role key anywhere; `GEMINI_API_KEY` never reaches the client bundle), and `npm audit` (one moderate transitive `postcss` advisory bundled inside `next`'s own deps — not exploitable here, no action taken). Full findings list (severity-ranked) given to the user before fixing; two were fixed this session, three low-severity/informational ones were reported but not code-changed (raw DB error messages in a couple of 500 responses; the transitive postcss advisory).
- **Fix 1 — SSRF via DNS rebinding (`src/app/api/ingest/route.ts`):** the prior `assertPublicHttpUrl()` only validated the URL's *hostname string* (blocked literal private IPs, `.local`/`.internal` suffixes) but never checked the IP `fetch()` would actually connect to — an attacker-controlled domain with a DNS record pointing at `169.254.169.254`/`127.0.0.1`/RFC1918 ranges sailed straight through. Added `undici` as a dependency and built `ssrfSafeDispatcher` — a custom `Agent` whose `connect.lookup` performs the ONLY DNS resolution used for the actual TCP connect, validates every resolved address (reused the existing IPv4/IPv6 private-range logic, refactored into `isBlockedIp()`), and rejects before a socket ever opens if any address is non-public. Wired in via `dispatcher: ssrfSafeDispatcher` on every `fetch()` call in `fetchReadable()` (covers the initial URL and every manually-followed redirect hop equally). TLS SNI/cert-hostname validation is untouched since Node derives those from the URL, not from `lookup`'s return value. Had to fix a bug mid-implementation: undici's connector calls `lookup` with `{ all: true }` for its own Happy-Eyeballs logic and expects an array back — my first version ignored that and broke every real HTTPS request (`Invalid IP address: undefined`) until `ssrfSafeLookup` was rewritten to honor both the `dns.lookup`-style `(hostname, callback)` and `(hostname, options, callback)` call shapes.
- **Fix 2 — unbounded Gemini-quota abuse via quiz sessions:** ingest already had `INGEST_DAILY_LIMIT`; quiz `finish`'s `gradeSession()` call (identical cost/quota class) had no cap at all — a single account could loop start/answer/finish indefinitely. Per the user's specified split: `INGEST_DAILY_LIMIT` lowered from 20→9 default, and a new independent `QUIZ_AI_DAILY_LIMIT`=1/day added, tracked via a new `quiz_sessions.ai_graded` boolean (migration added to both `schema.sql` and `schema-quiz-tables-only.sql`, NOT yet applied to the live Supabase DB — same manual-apply pattern as every other schema file here). Deliberately NOT a shared 10-call pool — quiz's 1 AI-grading slot is fully independent of ingest usage, so 9 ingests in one day can never starve the review flow of its guaranteed AI-graded session. Exceeding the quiz cap never blocks finishing — it silently reuses the pre-existing heuristic (keyword-overlap) grading path, same as when Gemini itself errors.
- **Bonus fix while touching the daily-cap logic:** the ingest cap's day boundary was computed with server-local `new Date().setHours(0,0,0,0)` — on Vercel that's effectively UTC midnight, not the user's midnight, inconsistent with `/api/plan` and `/api/quiz` finish (which both already use the `clampTz`/`localTodayForOffset`/`dayStartUtcIso` pattern from `dates.ts`). Fixed to match, threading a new `tz` param through `ingestText()`/`ingestLink()` → `/api/ingest`. Necessary anyway for the new popup (below) to show an accurate reset time.
- **New "daily limit reached" popup on `/add`** (user-specified UX): hitting the ingest cap now returns `resetAt` (ISO instant of the user's next local midnight) in the 429 body instead of just an error string. Added `ApiError` class to `data.ts` (carries HTTP status + `resetAt` so the UI can branch on 429 without parsing message text) and a new modal on `/add` — styled identically to the existing demo-gate popup — showing "You can add more starting tomorrow at [time]".
- **Fix 3 — curl/direct-access hardening:** live-curled every API route with no auth/cookies from a real terminal to verify none leak data or stack traces (confirmed clean — this local dev instance has no Supabase configured so routes 500 before reaching the auth check, but response bodies were empty in every case, and the real 401-before-any-data-access behavior was already independently verified by reading every route's code, where the auth check is unconditionally the first thing each handler does). Also tightened `PUBLIC_PATHS` matching in `src/proxy.ts` from a raw prefix match (`path.startsWith(p)`) to exact-or-segment match (`path === p || path.startsWith(p + "/")`) — not currently exploitable (no route collides with `/login`/`/auth` today) but a real gap for any future route sharing that text prefix.

## 2026-07-16 — Fix false-positive gibberish rejection on real link ingests
- **Why:** user reported `https://arpitbhayani.me/blogs/bm25` (a real, substantial technical blog post) got rejected by `/api/ingest` with "That doesn't look like real content to learn from" — the gibberish guardrail added earlier today (see the entry below) was too aggressive.
- **Root cause 1:** the repeated-pattern-flood regex in `looksLikeGibberish()` (`src/lib/guardrails.ts`) matched ANY 10+ char run of a repeated 1-4-char unit, including pure punctuation. The BM25 post renders its formulas as ASCII fraction bars (`──────────────────────────────────────────────`), which tripped it immediately — any technical writing with ASCII dividers/table borders/box-drawing rules would fail the same way. **Fix:** the flood check now only fires when the repeated unit contains a letter or digit; punctuation-only floods that really are spam (e.g. `>>> >>> >>>` quote-chain garbage) are still caught by the word-shape check below (none of their "words" contain a letter).
- **Root cause 2 (more severe, would have rejected almost any real article):** the distinct-character-ratio check assumed real prose has "real variety" in its character distribution — but English only has 36 possible alnum characters (26 letters + 10 digits), so that ratio mathematically saturates and collapses toward 0 for ANY text past roughly 300 characters (~2 sentences), independent of repetition. Verified against unrelated control paragraphs (a Wikipedia-style paragraph on the Roman Empire, a photosynthesis paragraph) — both tripped it too. Its intended purpose (catching short repeating alphabets) was already redundantly covered by the flood-regex and the distinct-word-ratio check, so it was **removed entirely** rather than re-tuned.
- **Verified:** replayed the actual `fetchReadable()` HTML→text extraction against the real page (curled it, ran the exact route.ts regex pipeline in a scratch script) to reproduce the bug outside the app, then ran the real `src/lib/guardrails.ts` (via `tsx`, not a reimplementation) against a 12-case suite: all original spam/mash cases (keyboard mash, `hahaha`, `asdasd`, quote-flood, symbol spam, repeated-word spam) still correctly rejected; the real BM25 blog, short/medium/long real prose, a markdown-divider+prose mix, and a JS snippet all correctly pass now. `tsc --noEmit` and `npm run build` clean.
