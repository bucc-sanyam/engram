# Knovis — Project Summary

> **Purpose of this file:** single source of truth for new sessions. Read this + `SESSION_LOG.md` + `TASKS.md` first; do NOT scan the repo unless something here is insufficient or contradicted. Keep this file updated after every meaningful change.

## What it is
"Your second brain" — a learning app. Log what you learn, see it as a living 3D knowledge graph, and get AI-scheduled quizzes before you forget. Gamified (daily streak). Runs in **demo mode** without sign-in (DEMO badge, seeded topics).

## Stack (fixed — do not propose alternatives)
- **Next.js 16.2.10** (App Router, Turbopack) + React 19.2.4 + TypeScript 5
- **Tailwind v4** (in `dependencies`, not dev — needed for prod builds)
- **Supabase** — auth (email; Google OAuth was removed) + Postgres DB (`@supabase/ssr`, `@supabase/supabase-js`) + **pgvector** for RAG. **Decision (2026-07-15): stay on Supabase, do NOT migrate to TiDB** — auth + RLS + browser-direct queries are load-bearing and TiDB has neither; when 500MB is hit, upgrade to Pro ($25/mo, 8GB, no 7-day pause) rather than migrate.
- **Gemini** via `@google/genai` — quiz generation / ingest / planning + **embeddings** (`gemini-embedding-001`, 768-dim) for RAG. **Model fallback chain** in `gemini.ts`: on 429/quota or 503/overload the call retries down `GEMINI_MODEL` → `GEMINI_MODEL_FALLBACKS` (default flash → flash-lite → 2.0-flash → 2.0-flash-lite); a 429'd model is skipped for 5 min per warm instance.
- **three.js 0.185** — the 3D brain scene
- **Deploy target: Vercel** (`render.yaml` was removed); Supabase free tier for db+auth

## Design language
Dark, organic, atmospheric — NOT boxy/generic. Mesh glows, blobs, pill shapes, organic radii. Fonts: Inter (body), **Space Grotesk** (display + canvas labels), JetBrains Mono (tiny caps labels), **Fraunces** (`--font-serif`, long-form reading on topic blog pages — see `.article-lead`/`.article-body` in globals.css). Ambient blob divs in `layout.tsx`. Background `#0b0a0e`.

**Paper Mode** (reading-view light theme, `/blogs/*`): toggled by a floating bottom-right button (`PaperModeToggle`) inside `ReadingThemeProvider` (`src/context/ReadingThemeContext.tsx`), which toggles the `.paper-mode-active` class + a canvas transition (`ThanosSnapCanvas.tsx` — a "paper-snap": small paper shards rotate/flip away shedding ash particles, sweeping radially from the button). Recolor is a large `!important` cascade in `globals.css` (token refactor still pending — see TASKS). **Accent legibility on the cream page is handled in components, not CSS:** bright per-topic pastel accents are darkened at render time via `src/lib/viz-theme.ts` (`darkenForPaper`/`tintForPaper`/`useVizPalette`/`useAccentColor`) — consumed by all 4 `viz:*` primitives and `src/components/AccentText.tsx` (`AccentText`/`DifficultyPill`/`AccentPill`, used on DSA question/chapter pages). Reason: inline arbitrary hexes can't be enumerated by `[style*="color:#…"]` CSS remaps.

**Branding (renamed from Engramia 2026-07-15):** the app is **Knovis** everywhere — knowledge + vision, deliberately echoing "novice" (the About page's origin story leans into this: every real learner stays a novice). The meaning tagline "kno·vis — knowledge meets vision" sits under the wordmark in `Nav.tsx`, on the login page, in layout metadata, and in the footer. Guest cookie is `knovis_guest`; localStorage/sessionStorage keys use a `knovis.*` prefix (tour state, demo state, notes) — renaming these was a deliberate part of the rebrand, so pre-existing guest sessions/localStorage from the Engramia era reset silently (no migration, no errors).

## AI-call budget (deliberate design — keep it this way)
Exactly **1 Gemini call per ingest** and **1 per finished quiz session**; everything else is zero-AI:
- **Ingest** (`/api/ingest`): the single `extractKnowledge()` call returns a **question bank** (7-8 per topic: ~2 open, ~1 quickfire, ~2 MCQ (4 options + correct_index), ~1-2 **truefalse** (options ["True","False"] + correct_index), ~1 **multi** (4-5 options + correct_indices array); each with model_answer + difficulty) and **facts** (1-2/topic) → stored in `questions` & `facts` tables. Flashcards are GONE (no extraction, no table writes).
- **Quiz** (`/api/quiz` actions): `start` picks bank questions (least-asked, review-count-matched difficulty, kind rotation open→mcq→truefalse→quickfire→multi; synthesized local fallback for pre-bank topics) into a `quiz_sessions` snapshot (client never sees answers); `answer` upserts each answer into `quiz_answers` as submitted (multi picks are JSON-serialised into the `answer` text column — no schema change); `finish` grades choice kinds (mcq/truefalse exact, multi with partial credit: exact=5, else 0-4 by hits−wrong over key size) deterministically + ONE `gradeSession()` batch call for typed answers, gated at `QUIZ_AI_DAILY_LIMIT`=1/day (keyword-overlap fallback if Gemini fails OR the daily cap is already spent, call skipped entirely if nothing typed) → report card (score_pct, summary, strengths/focus, per-item full Q&A) saved on the session (idempotent, `ai_graded` flag set iff `gradeSession()` actually ran), SRS per topic from avg score, reviews rows, bank usage bumped. The old `rate` (flashcard) action is removed. **Since 2026-07-21 `/recall` starts one session per story-series group, so a full run finishes several sessions → up to N `finish` calls. The AI-grading budget is now per-user dynamic: `QUIZ_AI_DAILY_BASE` (env `QUIZ_AI_DAILY_LIMIT`, default 1 = the General section) + one slot per row in `user_stories` (started stories) — so every section a user can submit gets a real AI-graded report; exceeding it still falls back to the heuristic grader.**
- **Plan** (`/api/plan`): Gemini narrative REMOVED — headline from day-rotating templates, insight reuses a `topic_links.reason` between today's topics (`composeNarrative()`).
- **Fact of the day**: `getFactOfTheDay()` in data.ts picks deterministically by day from `facts` (joined topic name; fallback = a topic key_point). Dashboard card, right column. Zero AI.

## RAG layer (added 2026-07-15, hardened 2026-07-15 — `supabase/schema-rag.sql` THEN `schema-rag-v2.sql` must both be run to activate)
User content is the source of truth. Everything ingested (article/notes) and every typed answer is chunked, embedded and stored so generation RETRIEVES the user's own material instead of leaning on the model's world knowledge. **Strictly additive + best-effort:** if either schema file isn't run yet, or embedding fails, every path degrades to the old behaviour (no errors).
- `src/lib/chunk.ts` — dependency-free overlapping chunker (~1400 chars ≈ 350 tokens, 200-char overlap, paragraph/sentence aware).
- `src/lib/gemini.ts` `embedTexts()/embedText()` — Gemini embeddings, L2-normalised client-side (truncated 768-dim isn't auto-normalised), `RETRIEVAL_DOCUMENT` vs `RETRIEVAL_QUERY` task types.
- `src/lib/rag.ts` — chunk/embed/store (`indexContent()`, `indexAnswers()`), retrieval (`retrieveContext()`, `formatContext()`), plus the v2 additions below. All catch-and-continue; recognise "RAG not installed" (42P01/42883) and stay quiet.
- **Wiring:** `/api/ingest` retrieves the user's prior related chunks and passes them to `extractKnowledge(text, existingTopics, priorContext)` as grounding (skipped on first ingest — this retrieval stays on the critical path, it shapes the prompt), then indexes the new entry. **Indexing itself runs in Next's `after()`, deferred until after the response is sent** (fixed 2026-07-15 — it was blocking the response and contributing to a 504 `FUNCTION_INVOCATION_TIMEOUT` on link ingests). `/api/quiz` finish indexes typed (open/quickfire) answers (still synchronous — quiz finish isn't the tight path link ingest is).
- Blog long-form generation from RAG is deliberately NOT wired (would add an AI call per page view, breaking the zero-AI-render budget) — see TASKS backlog.

### RAG v2 — document registry, index versioning, hybrid search, observability (added 2026-07-15, `supabase/schema-rag-v2.sql`)
Hardens the RAG layer against three gaps found on review: no dedup, vector-only retrieval, and zero observability. **Cross-encoder reranking was evaluated and deliberately deferred** (no ML runtime in this stack; a Gemini-based reranker would add a 2nd AI call per ingest, breaking the "exactly 1 Gemini call per ingest" budget) — see TASKS backlog.
- **Document registry + content-hash dedup:** `rag_documents` table (one row per indexed entry/answer, `content_hash` = sha256 via `node:crypto`). `findDuplicateEntry()` in `rag.ts` is checked in `/api/ingest` BEFORE the Gemini call — resubmitting identical text/URL now reuses the existing entry (200 + `duplicate:true` + friendly message, `xp:0`) instead of creating a duplicate `entries`/`knowledge_chunks`/`questions` row every time (a real bug that existed until now). `upsertDocument()` (private) makes `indexContent()` idempotent: re-indexing the same `entryId` deletes-then-replaces its chunks rather than duplicating. Partial unique index scopes the hash constraint to `source='entry' and status='active'` so answer chunks (which can legitimately repeat) aren't affected.
- **Alias-based index versioning:** `knowledge_chunks.index_version` + per-user `rag_index_pointers(user_id, active_version)` — the "alias" a reindex flips atomically. New ingests always tag chunks with the CURRENT `getActiveIndexVersion()`. `POST /api/rag/reindex` (new route, self-serve/RLS-only, no service-role key) builds a new generation in the background (`reindexDocument()`, paginated 200 docs/call, safe to call repeatedly) and only swaps the pointer + prunes the retired version once a full pass leaves zero pending documents — old version keeps serving every live retrieval throughout. Infrastructure for a future need (chunk-size/embedding-dim changes); nothing triggers it automatically today.
- **Hybrid search:** `match_knowledge_chunks_hybrid` SQL function — Reciprocal Rank Fusion (k=60) between vector kNN and Postgres `tsvector`/`ts_rank_cd` keyword search (the standard BM25 stand-in; true BM25 isn't native to Postgres). Query lexemes are OR'd (not AND'd like `plainto_tsquery`/`websearch_to_tsquery`) since the real caller passes a ~2000-char grounding excerpt, not a short phrase. `retrieveContext()` tries hybrid first, falls back to the (now version-aware) plain-vector `match_knowledge_chunks` via the existing `isMissingRag()` catch if the new SQL hasn't been applied yet.
- **Observability:** `rag_retrieval_logs` (chunk-level attribution: query, mode, index_version, per-chunk id/similarity/rank as jsonb — written by `retrieveContext()`, `entry_id` backfilled via `linkRetrievalLogToEntry()` once the ingest's entry exists, since retrieval runs before the entry does). `getRetrievalMetrics()` aggregates recent logs in JS (avg similarity, zero-hit rate, by-mode/by-day breakdown). `rag_index_version_correlation()` SQL function + `getIndexVersionCorrelation()` wrapper join retrieval logs → grounded topics → `reviews.score` by `index_version` — **explicitly correlation, not causal regression detection** (several steps removed: retrieval quality → extraction → question quality → the user's own recall). **No new UI** — these are queryable diagnostics only, matching how this app has no admin dashboard anywhere.
- `knowledge_chunks` table + HNSW cosine index + both match functions + `rag_documents`/`rag_index_pointers`/`rag_retrieval_logs`, all RLS-scoped to `auth.uid()`.

## Security posture (reviewed 2026-07-15, hardened 2026-07-18)
- **SSRF guard on link ingest, DNS-rebinding-safe (`src/app/api/ingest/route.ts`, fixed 2026-07-18):** `assertPublicHttpUrl()` is only the cheap string-level pre-filter (protocol, credentials, port, literal private IPs, `.local`/`.internal`). The REAL boundary is `ssrfSafeDispatcher` — a custom `undici` `Agent` whose `connect.lookup` resolves DNS itself, rejects the request if ANY resolved address is loopback/private/link-local/CGNAT, and only then hands the validated IP to the actual TCP connect (same lookup call does both, so there's no second, unvalidated DNS query for an attacker's server to answer differently). Closes the bypass where a hostname string looks harmless but its DNS record points at `169.254.169.254`/`127.0.0.1`/internal ranges — the old hostname-only check couldn't catch that. TLS SNI/cert hostname validation is unaffected (Node derives those from the URL, not from what `lookup` returns). Verified against real DNS-rebinding test domains (`*.nip.io`, which resolves to whatever IP is embedded in the subdomain) — metadata/loopback/private targets all blocked, real public HTTPS sites (incl. TLS) still work.
- **Per-user daily Gemini-call caps, split 9 + 1 (`INGEST_DAILY_LIMIT`=9, `QUIZ_AI_DAILY_LIMIT`=1, added 2026-07-18):** ingest capped at 9/day (down from an unbounded-in-practice 20); quiz AI grading (`gradeSession()`) separately capped at 1/day via a new `quiz_sessions.ai_graded` flag — deliberately NOT shared with the ingest budget, so a heavy ingest day can never fully starve the review flow. Exceeding the quiz cap never blocks finishing a session — it silently reuses the existing heuristic (keyword-overlap) grader, the same fallback already used when Gemini itself errors. Both day-boundaries now use the CLIENT's local timezone (`clampTz`/`localTodayForOffset`/`dayStartUtcIso` from `dates.ts`) — the ingest cap previously reset at the server's own midnight (effectively UTC on Vercel), not the user's. Hitting the ingest cap returns `resetAt` (ISO instant of the user's next local midnight) in the 429 body; `/add` shows a popup ("You can add more starting tomorrow at —") instead of just inline error text, via a new `ApiError` class in `data.ts` that carries the HTTP status so the UI can branch on 429 without parsing message text.
- **`PUBLIC_PATHS` exact/segment match, not prefix match** (`src/proxy.ts`, fixed 2026-07-18): was `path.startsWith(p)`, which would treat a hypothetical future route like `/login-help` as public just for sharing a text prefix with `/login`. Now `path === p || path.startsWith(p + "/")`.
- All API routes wrap `request.json()` in try/catch (malformed-body → 400). `/auth/callback` redirects to `/login?error=auth` when `exchangeCodeForSession` fails instead of silently landing on the dashboard.
- RLS: every table `own <x>` policy is `auth.uid() = user_id` (profiles: `= id`; entry_topics via entry ownership). data.ts queries Supabase directly from the browser — RLS is the only thing isolating users, so keep it on every new table (the new `knowledge_chunks` has it).
- **Ingest guardrails (added 2026-07-16, fixed 2026-07-16, `src/lib/guardrails.ts`):** `looksLikeGibberish()` — dependency-free heuristics (repeated-pattern runs, meaningful-char ratio, distinct-word ratio) reject obvious spam/keyboard-mashing before it burns a Gemini call or pollutes the knowledge base; deliberately lenient (dense technical text, code, LaTeX, non-English prose all pass — tested against a case suite). **Two false-positive bugs fixed same day** (found via a real link-ingest of an external blog post that got wrongly rejected): (1) the repeated-pattern-flood regex ran on ANY 10+ char repeat including pure punctuation, so ASCII fraction bars/table borders/box-drawing dividers common in technical writing tripped it — now requires the repeated unit to contain a letter/digit; (2) the distinct-character-ratio check (since removed entirely) was not scale-invariant — English has only 36 possible alnum characters, so ANY real prose past ~2 sentences (~300 chars) mathematically saturates the alphabet and its ratio collapses below the threshold regardless of legitimacy; its intended purpose was already redundantly covered by the flood-regex and the distinct-word-ratio check, so it was dropped rather than re-tuned. `sanitizeField()` strips control chars and clamps length, applied to EVERY field `extractKnowledge()` returns (title, summary, topic name/summary/key_points, connection reason, question prompt/model_answer/options, fact text) before DB insert; topic `category` is validated against `CATEGORY_COLORS` keys (`src/lib/types.ts`) with a `"General"` fallback — previously unvalidated. **Prompt-injection hardening** in `extractKnowledge()` (`gemini.ts`): the pasted/fetched text and the RAG grounding block are explicitly framed as "untrusted data, not instructions" with a rule telling the model to never follow embedded commands (relevant since link ingestion feeds arbitrary third-party webpage text into the prompt). Blast radius if a prompt injection ever succeeded is contained — no `dangerouslySetInnerHTML` anywhere in the app (verified by grep), so AI-generated text can only ever render as inert text via React's default escaping, with one exception below.
- **Markdown link scheme validation** (`src/components/Markdown.tsx`, used by `/notes`): `isSafeHref()` only renders `[text](url)` as a clickable `<a>` when the scheme is `http`/`https`/`mailto` (or scheme-less/relative) — a `javascript:`/`data:` URI now renders as inert plain text instead of an executable link. Verified live: a `javascript:` and a `data:text/html,<script>` link both render as non-clickable text while a normal `https://` link still works.

## Dates / "today" (fixed 2026-07-15)
- `src/lib/dates.ts` is the single source for day boundaries. The app's "day" is the USER'S LOCAL calendar day, never the server's UTC day. Client passes its tz offset (`new Date().getTimezoneOffset()`) to `/api/plan?tz=` and quiz finish/complete bodies; server uses `localTodayForOffset`/`dayStartUtcIso`/`clampTz`. Client bucketing (calendar, heatmap, review detail) uses `localDayKey(new Date(created_at))`. This was the root cause of "answered questions show as not attempted": a UTC-vs-local day mismatch (e.g. evening IST reviews landing on the next UTC date) plus done-tracking that only read `reviews`. `doneTopicsToday()` now unions `reviews` rows AND `topics.last_reviewed_at` (always written on finish) so a failed reviews insert can't strand items.
